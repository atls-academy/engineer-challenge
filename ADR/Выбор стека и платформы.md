# ADR-001: Выбор языка и платформы

**Статус:** Принято  
**Дата:** 2026-03-14  
**Контекст:** Engineer Challenge — модуль аутентификации (регистрация, авторизация, восстановление пароля)

---

## Решение

**Kotlin + JVM**

---

## Контекст и постановка проблемы

Необходимо реализовать архитектурно-зрелый модуль аутентификации с явным DDD, CQRS и IaC.  
Задача оценивает не скорость сборки, а качество инженерного мышления и обоснованность trade-offs.

Рассматривались три варианта: **Kotlin/JVM**, **Python**, **Go**.

---

## Рассмотренные варианты

### Kotlin/JVM (выбран)

**Плюсы:**
- Разработчик обладает экспертной квалификацией в Kotlin → ниже когнитивная нагрузка → фокус на архитектуре, а не на языке
- Нативная выразительность для DDD: `sealed class` для команд/событий/ошибок, `data class` для Value Objects, `value class` для обёрток примитивов
- Возможности ФП: `Either<DomainError, T>`, явный контроль сайд-эффектов
- Корутины: структурированная асинхронность с явным контекстом — важно (rate limiting, timeout, replay protection)
- Большое сообщество, зрелая экосистема, много разработчиков на рынке
- JVM: высокий перформанс под нагрузкой, широкие возможности настройки JIT/GC

**Минусы:**
- Высокое потребление RAM по сравнению с Go/Python
- Долгий холодный старт, быстро горизонтально отмасштабироваться не получится(либо сложные решения с GraalVm)

**Снижение рисков**
Для данного челленджа цель — локально воспроизводимый стенд, холодный старт не критичен.
В production-сценарии с требованием быстрого старта рассмотрел бы GraalVM Native Image или Quarkus.

---

### Python ❌ (отклонён)

**Плюсы:**
- Быстрый старт -> легко отмасштабироваться горизонтально
- хорошие возможности асинхронности
- низкие затраты на инстанс
- наличие ФП-возможностей

**Причины отклонения:**
- Отсутствие полноценной многопоточности — риски при высоконагруженных auth-флоу
- Более слабая система типов усложняет моделирование сложных доменных инвариантов
- Квалификация ниже, чем в Kotlin → выше риск архитектурных ошибок именно там, где задача требует максимальной аккуратности
- Меньше устоявшихся паттернов для DDD/CQRS в экосистеме

---

### Go ❌ (отклонён)

**Плюсы:**
- Быстрый запуск
- минимальное потребление ресурсов
- сильная статическая типизация
- хорошая многопоточности и асинхронность(горутины и каналы)

**Причины отклонения:**
- Нет опыта разработки на Go → высокий риск непредвиденных проблем в ходе реализации
- Минималистичная стандартная библиотека требует большего количества ручных решений
- Горутины могут создавать сложноотлаживаемые состояния гонки
- Минималистичная система типов затрудняет выразительное DDD-моделирование
- Отсутствие устоявшихся паттернов для CQRS/DDD в экосистеме

## Последствия

- Фреймворк: **Ktor** или **Spring Boot** — решение будет принято в следующем ADR
- Transport: **gRPC** (grpc-kotlin) как основной протокол согласно требованиям челленджа
- FP: **Arrow** для моделирования доменных ошибок и контроля сайд-эффектов
- Async: Корутины для всех I/O операций
- В production при требовании низкого холодного старта → GraalVM Native Image
